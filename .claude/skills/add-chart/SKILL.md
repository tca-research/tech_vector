# Add a new chart

Every chart in this repo is generated by `scripts/charts/build_charts.py`
from a `CONFIG` dict + a `load_*_data()` loader, registered as one entry in
the `CHARTS` list at the bottom of that file. There is no chart library —
each output is a self-contained HTML file with inlined CSS/JS.

## 1. Decide: reuse a chart type, or add a new one

Most new charts are a **new dataset in an existing shape**, not new code:

- A category + single value comparison → `verticalBar` or `rankedBar`
- A category + N series over time → `line`, `smallMultiples`, `stackedSmallMultiples`
- A KPI vs. target → `gauge` (now a flat KPI card, not a needle)
- A ranked top-N list → `rankedBar` or `topRankedDual`

If your data fits one of the ~14 existing types, skip straight to step 2 —
you only need a new `CONFIG` + loader + `CHARTS` entry, no new HTML/JS/CSS.

Only add a genuinely new visual type (step 4) when the interaction pattern
itself is new. If so, consult the **dataviz** skill first for form/color/
interaction choices before writing markup.

## 2. Add a CONFIG dict + loader in `build_charts.py`

Add a new numbered section (matching the existing `# === N. NAME ===` /
`# =====` banner style) with:

- A `*_CONFIG` dict: `pageTitle`, `title`, `caption`, `source` (use a
  `"{year}"` placeholder if the source is "current as of last fetch" data
  like WGEA/ABS/OECD headline pages; omit it for a source citing a fixed
  historical vintage, e.g. a specific report year that will never change).
- A `load_<name>_data(config=..., source_file="...")` function reading a
  CSV from `data/output/` or `data/input/manual_pull/`, that:
  - **Re-sorts by parsed date** — never trust a CSV's row order, even when
    it looks sorted (see `DATA_FORMAT.md`, "Rule zero").
  - **Converts any date column to ISO** (`"YYYY-MM-DD"` or `"YYYY-MM"`)
    immediately after loading, and calls `assert_iso_date()` on it. A
    non-ISO date (e.g. ABS-style `"Aug-06"`) parses in JS with no error and
    silently lands at the wrong x-position — this exact bug has shipped
    before.
  - Returns a list of dicts matching the JSON shape your chart's JS expects.

## 3. New HTML/JS/CSS assets (only if step 1 needs a genuinely new type)

In `scripts/charts/assets/`:

- `<name>.html` — body markup: a `<figure class="chart-card">` wrapping a
  title element, a mount point for the chart, and a `.tooltip` div if
  interactive.
- `<name>.js` — reads `DATA.<key>.config` / `DATA.<key>.data` at render
  time, calls `applyChromeConfig(cfg)` for title/caption/source-line/
  download-button wiring, renders via plain SVG/DOM.
- Any new shared CSS goes in `shared.css` — **this file is inlined into
  every single chart's HTML**, so scope new class names to your chart
  specifically (e.g. `.status-row`, not `.row`) to avoid colliding with an
  existing chart's styles.

## 4. Register in the `CHARTS` list

```python
{
    "key": "yourKey",          # shared by all charts using the same body/js template
    "body_file": "yourname.html",
    "js_file": "yourname.js",
    "loader": load_your_name_data,
    "config": YOUR_CONFIG,
    "out_file": "your_output_name.html",   # final charts/<name>.html filename
},
```

If your loader needs the current build timestamp (rare — only
`references` and `status` do today, for "viewed as of" / "next update"
dates that must reflect *today*, not the last data change), extend `main()`'s
special case:

```python
elif chart["key"] in ("references", "status", "yourKey"):
    data = chart["loader"](chart["config"], now=now)
```

## 5. Build and verify

```bash
python3 scripts/charts/build_charts.py
```

- Read the console output: it should say `data changed, stamping today's
  date` for your new chart. Run it again with no CSV changes — it should
  say `data unchanged, keeping previous date` (confirms the freshness
  mechanism sees your loader's output correctly).
- Headless-Chrome console-error check:
  ```bash
  CHROME="/Applications/Google Chrome.app/Contents/MacOS/Google Chrome"
  "$CHROME" --headless=new --disable-gpu --no-sandbox --virtual-time-budget=3000 \
    --enable-logging=stderr --v=1 --window-size=900,600 \
    --screenshot=/tmp/newchart.png "file://$(pwd)/charts/your_output_name.html" 2>&1 \
    | grep -E ":CONSOLE"
  ```
  Grep the output for `error`/`exception`/`uncaught` — should be empty.
- **Actually view `/tmp/newchart.png`** (Read tool) — a clean console log
  doesn't catch layout bugs (label overlap, wrong colors, misaligned rows).
- Run the same console-error sweep across *all* `charts/*.html`, not just
  the new one — a `shared.css`/`shared.js` change ripples into every chart
  since they're inlined into each output file.

## 6. Document the contract

Add a new entry to `scripts/charts/DATA_FORMAT.md`'s "Per-chart data
contracts" section, matching the existing format: the JSON shape your
loader returns, plus prose notes on any non-obvious behavior (derived
fields, special-cased dates, units that must match, etc.).

## Gotchas worth re-reading before you start

- **ISO dates only** — see step 2.
- **Gauge/KPI met-vs-not status is derived**, not hand-entered:
  `parse_metric_value()` strips `$`/`%`/`k`/`M`/`B` and compares
  `current_label` vs `target_label` as plain numbers. Mismatched units
  (current in `k`, target in `M`) silently produce the wrong pill.
- **The "last updated" date only advances when data (or the source
  template's wording) actually changed** — `read_previous_chart_state()` +
  `data_unchanged` in `main()`. Don't expect every build to bump every
  date.
- `automated_data_prep.py`'s pay-quartiles output uses unseeded
  `random.sample()` for example-role text — running it just to verify
  (not because real data changed) produces a spurious diff. Don't commit
  that noise; `git checkout --` it if you weren't intending a real change.
